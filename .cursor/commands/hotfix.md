# /hotfix

## 개요
프로덕션 장애 대응을 위한 긴급 수정 파이프라인입니다.
/impact(축약) → /implement → /test(최소) → /pr을 일괄 실행하여 5분 내 완료를 목표로 합니다.
정상 개발 프로세스를 일부 스킵하므로 주의가 필요합니다.

## 입력
- 장애 설명 (예: "입고 목록 조회 API 500 에러")
- 긴급도 (P1/P2)

## 워크플로우

### Step 0: 긴급도 판단 (30초)
- **P0/P1:** 즉시 hotfix 진행
- **P2:** 일반 /implement 프로세스 권장

### Step 1: Impact 분석 (축약) (1분)
- /impact 실행 (패턴 매칭만)
- 영향받는 파일 빠른 식별
- 위험도 평가 (🔴만 중점)
- 회귀 테스트 범위 최소화

### Step 2: 긴급 구현 (2-3분)
- /implement 실행 (최소 변경만)
- 한 파일, 한 함수 기준
- 코드 리뷰 스킵 (사후 리뷰로 변경)
- git commit -m "🚨 Hotfix: {설명}"

**제한:**
- DB 스키마 변경 금지
- 신규 기능 추가 금지
- 기존 로직만 수정

### Step 3: 최소 테스트 (1-2분)
- /test 실행 (최소 테스트 케이스만)
- 수정된 기능만 검증
- 회귀 테스트 스킵 (사후 full test로 변경)
- 로그 확인 (에러 메시지, 성능)

**테스트 항목:**
- [ ] 변경된 엔드포인트 응답 200?
- [ ] 관련 기능 정상 동작?
- [ ] 에러 메시지 명확한가?

### Step 4: PR 제출 (1분)
- /pr 실행
- PR 제목: `🚨 Hotfix: {설명}`
- 본문: 간단한 설명 + 테스트 결과
- 리뷰어: @on-call-engineer
- **Merge 조건:** 1명 승인 + CI 통과

### Step 5: 모니터링 (지속)
- 배포 후 5분 모니터링
- 에러율, 응답시간 확인
- 재발 여부 확인

### Step 6: 사후 처리 (24시간 내)
- 전체 /review 실행
- 전체 /test 실행
- 코드 개선 PR 제출
- 근본 원인 분석 (RCA) 작성

## 산출물
- **hotfix-{date}.md**
  - 장애 설명
  - 영향도 (영향받는 사용자, 기간)
  - 수정 내용
  - 테스트 결과
  - 재발 방지 계획

- **PR**
  - 제목: `🚨 Hotfix: {설명}`
  - 간단한 설명
  - 테스트 결과

## 체크포인트
- [ ] 정말 긴급한 장애인가? (P0/P1?)
- [ ] DB 스키마 변경이 없는가?
- [ ] 최소한의 변경만 했는가?
- [ ] 최소 테스트를 통과했는가?
- [ ] PR 제목에 🚨이 있는가?
- [ ] 배포 후 모니터링을 했는가?
- [ ] 사후 처리 일정을 계획했는가?

## 주의사항
- **Hotfix는 예외 상황입니다.** 정상 프로세스를 건너뛰면 안 됩니다.
- 배포 후 **반드시 24시간 내에 전체 테스트 + 리뷰를 진행하세요.**
- Hotfix 메시지(🚨)는 본 명령어 사용 시에만 사용하세요.
- 근본 원인 분석(RCA) 없이 같은 장애 반복 금지
- 팀장 알림 (Slack, Email) 필수

## 타임라인 예시
```
T+0:00 장애 보고
T+0:30 긴급도 판단 (hotfix 결정)
T+1:00 영향도 분석 완료
T+3:00 구현 + 최소 테스트 완료
T+4:00 PR 제출 + 1명 승인
T+4:30 배포 완료
T+4:35 모니터링 시작
T+5:00 정상화 확인
T+25:00 사후 full test 완료
T+26:00 코드 개선 PR 제출
```
