# /gen-req

## 개요
RFP, 인터뷰 결과서, 현행시스템 분석서를 기반으로 명확하고 구체적인 요구사항정의서를 작성합니다.
추측이나 가정 없이, 불명확한 부분은 반드시 사용자에게 질의하여 확인 후 진행합니다.
분석 작업만 수행하며 코드 생성은 하지 않습니다.

## 입력
- **RFP(PRD)**: `docs/01.analysis/01.rfp/*_prd.md` (해당 경로의 PRD 마크다운 파일. 복수 시 대상 파일을 명시하거나 모두 검토)
- **추가 요구사항(선택)**: `docs/01.analysis/02.requirement/*.md` — Step 2에서 수집하는 보완·명확화 내용(파일 여러 개 가능, 파일명 자유)
- 시스템명 (예: "wms")
- 시스템 전체명 (예: "WMS 창고관리시스템")

## 워크플로우

### Step 1: 요구사항 파악 (Critical)
- **목표**: 요구사항 문서 확보 및 기능 범위 명확화
- `docs/01.analysis/01.rfp/*_prd.md` 파일을 읽어 RFP(PRD) 내용을 확보한다.
- 시스템 맥락 파악 및 대상 기능 정의
- ⚠️ 위 패턴에 해당하는 파일이 없거나 비어 있으면 작업 중단 → PRD 작성·배치 후 재실행 요청

### Step 2: 추가 요구사항 수집 (Critical)
- **목표**: PRD만으로 부족한 정책·구현 방식·범위를 문서로 확정한다.
- **`docs/01.analysis/02.requirement/`** 폴더에 **`.md` 파일을 추가**하여 추가 요구사항을 수집한다. (인터뷰 메모, 정책 결정, UI·권한·배치 조건 등)
- 실행 시 `docs/01.analysis/02.requirement/` **이하의 모든 `.md` 파일**을 읽어 추가 요구사항으로 반영한다(하위 폴더 포함).
- ⚠️ 문서에 모순·누락이 있으면 **추가 `.md`로 보완**하거나, 불가피할 때만 사용자에게 확인한다. 추측·가정으로 빈칸을 메우지 않는다.
- 추가 `.md`가 **없으면** PRD만을 입력으로 Step 3로 진행한다.

### Step 3: 요구사항정의서 작성 및 검증
- **목표**: PRD + Step 2 추가 요구사항(`.md`) + 필요 시 사용자 확인 내용을 반영한 요구사항정의서 생성
- **저장 경로**: `docs/01.analysis/02.requirements/{fileName}.md`
- 기존 요구사항정의서 참조하여 검토 및 보완

## 요구사항정의서 템플릿 (8-Section)

```markdown
# [요구사항명]

**문서 버전**: v1.0
**생성일자**: [YYYY-MM-DD]
**담당자**: [관련분야] PL
**시스템**: {SYSTEM_FULL_NAME}
**메뉴 경로**: {1depth} > {2depth} > {3depth}

---

## 1. 개요
### 1.1 목적
### 1.2 범위

## 2. 핵심 기능 요구사항
### 2.[1-n] [요구사항명] ([요구사항ID])
#### 2.[1-n].1 기본 기능
#### 2.[1-n].2 비즈니스 규칙

## 3. 명확화된 정책 및 규칙
### 3.[1-n] [질문내용]
- [선택한 답변 정리]
**구현방안**: [시스템 관련 구현방안]

## 4. 예외 처리 및 오류 시나리오
### 4.[1-n] 예외 상황
- 오류 → "메시지" (오류코드 E0000~E9999)

## 5. 테스트 시나리오
### 5.1 정상 시나리오
### 5.2 예외 시나리오

## 6. 성능 및 보안 요구사항
### 6.1 성능 요구사항
### 6.2 보안 요구사항

## 7. 시스템 연동 요구사항
### 7.1 필수 연동 시스템

## 8. 데이터 모델 요구사항 [물리 모델 생성 금지]
### 8.1 핵심 엔티티
### 8.2 데이터 운영 고려사항
```
