# /review

## 개요
git diff를 수집하여 5대 관점에서 코드 리뷰를 수행합니다.
심각도별로 코멘트를 작성하여 PR 제출 전 품질을 보증합니다.

## 입력
- git log range (예: "main..HEAD")
- 또는 최근 커밋 수

## 워크플로우

### Step 1: Git Diff 수집
- 현재 브랜치 vs main 비교
- 변경된 파일 목록 추출
- 각 파일별 diff 수집
- 라인 수, 추가/삭제 통계

### Step 2: 5대 관점 검사

#### 관점 1: 아키텍처 & 설계 🏗️
- 계층 분리 준수 (DB/BE/FE)
- 디자인 패턴 (MVC, Composable)
- 순환 의존성 확인
- 코드 재사용성 (DRY)

**체크리스트:**
- [ ] Controller에 비즈니스 로직이 없는가?
- [ ] Service에 모든 비즈니스 로직이 있는가?
- [ ] FE가 직접 DB 접근하지 않는가?
- [ ] 공통 코드가 재사용되는가?

#### 관점 2: 보안 🔐
- SQL Injection (PreparedStatement 사용?)
- XSS (HTML 이스케이핑?)
- 인증/권한 체크 (필요한가?)
- 민감 데이터 (비밀번호, API 키 노출?)

**체크리스트:**
- [ ] SQL이 CDATA로 이스케이프되었는가?
- [ ] 동적 SQL에 PreparedStatement가 사용되었는가?
- [ ] API 엔드포인트에 인증이 있는가?
- [ ] 비밀번호가 평문으로 저장되지 않는가?

#### 관점 3: 성능 ⚡
- N+1 쿼리 문제
- 불필요한 루프/재계산
- 메모리 누수
- 대용량 데이터 처리

**체크리스트:**
- [ ] 반복문 내에서 DB 쿼리를 하지 않는가?
- [ ] 인덱스가 설정되었는가?
- [ ] 페이징이 구현되었는가?
- [ ] 비효율적인 JOIN이 없는가?

#### 관점 4: 네이밍 & 가독성 📝
- 변수명, 함수명 의도가 명확한가?
- 일관된 네이밍 규칙 (camelCase, snake_case)
- 코드 주석 (복잡한 로직에만)
- 매직 넘버 제거

**체크리스트:**
- [ ] 변수명이 자명한가?
- [ ] 함수명이 동사 + 목적어 형식인가?
- [ ] 일관된 네이밍 규칙을 따르는가?
- [ ] 매직 상수가 상수로 정의되었는가?

#### 관점 5: 에러 처리 & 로깅 🛡️
- try-catch 사용 (적절한 위치?)
- 예외 타입 구분
- 로깅 레벨 (DEBUG/INFO/WARN/ERROR)
- 사용자 친화적 에러 메시지

**체크리스트:**
- [ ] 예외가 처리되었는가?
- [ ] 에러 메시지가 사용자 친화적인가?
- [ ] 로깅이 적절한 레벨로 기록되는가?
- [ ] 스택 트레이스가 숨겨지는가? (XSS 방지)

### Step 3: 심각도별 코멘트 작성

- 🔴 **Critical (높음):** 보안, 데이터 손실, 아키텍처 위반
- 🟡 **Medium (중간):** 성능, 가독성, 표준 위반
- 🟢 **Low (낮음):** 스타일, 개선 제안

### Step 4: 자동 수정 제안
- 변경 전/후 코드 스니펫 제공
- 적용 가능한 항목은 자동 수정 (선택)

## 산출물
- **review-{branch}-{date}.md**
  - 요약: 파일 수, 라인 수, 이슈 수
  - 파일별 리뷰:
    ```
    ### backend/src/main/java/service/InboundService.java

    🔴 [Critical] Line 42: SQL Injection 위험
    ❯ updateStatus() 메서드에서 동적 SQL이 사용됩니다.
    ❯ 변경 전: String sql = "UPDATE ... WHERE id = " + id;
    ❯ 변경 후: PreparedStatement 또는 Mapper XML 사용

    🟡 [Medium] Line 15: N+1 쿼리 패턴
    ❯ 루프 내에서 DB 조회가 발생합니다.
    ❯ 제안: findAll() 메서드로 배치 조회

    🟢 [Low] Line 100: 변수명 개선
    ❯ int x → int itemCount
    ```

  - 요약 테이블:
    ```
    | 관점 | 이슈 수 | 심각도 |
    |------|--------|-------|
    | 보안 | 1      | 🔴   |
    | 성능 | 2      | 🟡   |
    | 아키텍처 | 0 | - |
    | 가독성 | 1 | 🟢 |
    | 에러처리 | 0 | - |
    | **합계** | **4** | - |
    ```

## 체크포인트
- [ ] 모든 계층(DB/BE/FE)이 검사되었는가?
- [ ] 보안 이슈가 없는가?
- [ ] 성능 우려사항이 해결되었는가?
- [ ] 코멘트가 명확하고 실행 가능한가?
- [ ] 자동 수정 제안이 정확한가?

## 주의사항
- 리뷰는 PR 제출 전 항상 수행
- 🔴 이슈는 반드시 수정 후 PR 제출
- 🟡 이슈는 개선 항목으로 기록 후 PR 제출 가능
- 🟢 이슈는 참고용이므로 필수 수정 아님
