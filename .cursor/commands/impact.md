# /impact

## 개요
변경 사항의 영향도를 체계적으로 분석합니다. 실무에서 가장 빈번하게 사용되는 명령어입니다.
DB 변경 시 자동으로 관련 Mapper, Service, Controller, FE까지 추적합니다.

## 입력
- 변경점 설명 (예: "twms_inbound 테이블에 'status' 컬럼 추가")
- 또는 수정된 파일 목록

## 워크플로우

### Step 1: 변경점 식별
- 수정 대상 파일 (DB/BE/FE)
- 변경 유형: 추가(+), 수정(~), 삭제(-)
- 변경 범위: 단일 엔티티 vs 다중 엔티티

### Step 2: 영향도 매트릭스 생성
- 행: 변경점 (테이블, 컬럼, 메서드 등)
- 열: 영향받는 파일 (계층별)
- 셀: 변경 유형 × 위험도 (🟢 낮음, 🟡 중간, 🔴 높음)

**예시:**
```
변경점: twms_inbound.status 추가
┌─────────────────────┬──────────────┬─────────────┬───────────────┐
│ 파일                 │ 변경 유형    │ 위험도      │ 이유          │
├─────────────────────┼──────────────┼─────────────┼───────────────┤
│ Mapper XML          │ + SELECT절   │ 🟡 중간    │ CDATA 필수    │
│ Service            │ + 유효성     │ 🟡 중간    │ 비즈니스로직  │
│ Controller         │ 변경 없음    │ 🟢 낮음    │ DTO 호환     │
│ FE 폼              │ + 필드       │ 🔴 높음    │ 검증 추가     │
└─────────────────────┴──────────────┴─────────────┴───────────────┘
```

### Step 3: 회귀 테스트 범위 결정
- 영향받는 계층의 모든 기능 테스트 목록
- 우선순위별 테스트 순서
- 예상 테스트 시간

### Step 4: 작업 순서 제안
- 의존성 기반 작업 순서 (Bottom-Up)
- 병렬 처리 가능 부분 식별
- 각 단계별 예상 시간 (T-shirt: S/M/L)

### Step 5: 문서화
- impact-{feature}-{date}.md 생성

## 산출물
- **impact-{feature}-{date}.md**
  - 변경점 요약
  - 영향도 매트릭스 (테이블)
  - 영향받는 파일 목록 (계층별)
  - 회귀 테스트 체크리스트
  - 작업 순서 및 예상 시간

## 체크포인트
- [ ] DB 변경 시 Mapper XML이 누락되지 않았는가?
- [ ] 영향받는 Service 메서드가 모두 식별되었는가?
- [ ] FE 컴포넌트의 수정 범위가 명확한가?
- [ ] 회귀 테스트 범위가 합리적인가?
- [ ] 작업 순서가 의존성을 만족하는가?

## 주의사항
- DELETE/TRUNCATE는 🔴 높음 위험도
- FK 변경은 모든 관련 테이블 확인 필수
- API 응답 구조 변경 → FE Composable 전체 검증
