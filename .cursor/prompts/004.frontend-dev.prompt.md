# OOO 시스템 - 프론트엔드 개발 

**⚠️ Task 정의서와 UI 설계서 미 첨부시 작업을 중단하고 사용자에게 첨부요청, 첨부된 파일은 누락없이 모두 파악**

> [!CRITICAL]
> **Role**: Senior Vue 3 & Element Plus Frontend Developer  
> **Goal**: Task 정의서와 UI 설계서를 완벽하게 준수하는 Frontend 소스코드를 생성한다.  
> **Constraint**: UI설계서의 좌측 화면의 번호(circle-number css class)와 오른쪽 설명영역과 설명을 위한 스크립트는 제거하고 구현합니다.
> 
> **필수 준수 문서**:
> - **개발 지침**: `.cursor/rules/frontend.dev.instructions.mdc` - 아키텍처, 기술 스택, 디렉토리 구조
> - **공통 라이브러리**: `.cursor/rules/frontend.composables.utils.instructions.mdc` - Composables, Utils, 표준 페이지 패턴
> - **UI/UX 가이드**: `.cursor/rules/frontend.uiux.instructions.mdc` - 디자인 시스템
>
> **UI 설계서의 API URL 은 생성된 Frontend 코드에 맞게 완벽하게 변환되어야 합니다.**
> **UI 설계서의 UI 와 완벽하게 일치하도록 Frontend 코드가 생성 되어야 합니다.**
> **UI 설계서의 기능 설명과 완벽하게 일치하도록 Frontend 코드가 생성 되어야 합니다.**
> **Backend 개발은 별도 작업으로 진행되므로 Frontend 개발만 하고 Backend 개발을 자동으로 진행하지 않습니다.**

## 필수 작업 계획 수립

**use plan_task tool** - 다음 **5단계**로 작업을 분할하여 계획 수립, 파일에 기록되지 않는 단계별 작업 결과는 최대로 요약해서 표시(LLM 사용료 절약), 각 Task 는 사용자가 확인 후 실행(자동 실행 금지)

1. **Task 정의서 분석 및 Mock Data 이관**: Task 정의서에서 메뉴 구조 파악, UI 설계서의 JSON 데이터를 `frontend/api/v1/` 구조로 이관
2. **페이지 분할**: UI 설계서 HTML을 표준 페이지 패턴(List, Edit 등)과 Composables를 사용하여 분할
3. **소스코드 연동**: Task 정의서 기반 메뉴 구성, Vue Router 설정, 컴포넌트 등록, 네비게이션 연동
4. **검토 및 보완**: UI/Task 명세 대조, 개발 지침 준수 확인, 체크리스트 검증
5. **사용자 테스트**: 브라우저 테스트 요청, 피드백 반영, 결과 기록

## 실행 프로세스

### 1단계: UI 설계서의 Mock Data 이관 및 Task 정의서 분석
**목표**: UI 설계서에서 사용하는 Mock 데이터를 프로젝트 Frontend 구조로 이관하고, Task 정의서를 분석하여 메뉴 구조를 파악

**수행 방법**:
- **Task 정의서 분석 (메뉴 구조 파악)**:
  - 첨부된 Task 정의서 파일들을 읽어 다음 정보 추출:
    - **메뉴 경로**: 각 Task 정의서의 "메뉴 경로" 필드 확인 (예: `사용자관리 > 회원가입/인증`)
    - **사용자 유형**: 각 Task 정의서의 "사용자 유형" 또는 "접근 권한" 섹션 확인
    - **메뉴 구조 도출**: Task 정의서들을 분석하여 사용자 유형별로 접근 가능한 메뉴 목록 생성
      - 메뉴 경로의 상위 분류(예: "사용자관리", "차량관리")를 메뉴 그룹으로 구성
      - 각 메뉴 그룹 하위에 해당하는 Task들의 메뉴 경로를 하위 메뉴로 구성
  - 메뉴 구조 분석 결과를 메모하여 3단계에서 `AppLayout` 컴포넌트 생성 시 활용
- **Mock Data 파일 확인**: UI 설계서 HTML에서 로드하는 API 경로 파악 (예: `../common/api/v1/brand-masters.json`)
- **데이터 이관**: `docs/02.design/02.ui/common/api/v1/[menu-name].json` → `frontend/api/v1/[menu-name].json`으로 복사
- **경로 검증**: 이관된 Mock 데이터 파일의 경로와 스키마 구조 확인
- **데이터 정합성**: Mock 데이터의 필드명이 Backend DB 스키마(`database/schemas/create_tables_phase1_2.sql`)와 일치하는지 확인

### 2단계: UI 설계서 html 을 프로젝트의 Frontend 구조와 지침에 맞게 분할

**목표**: UI 설계서의 단일 HTML 파일을 Frontend 개발 지침에 따라 컴포넌트와 페이지로 분리하여 재사용 가능하고 유지보수하기 쉬운 구조로 변환

**수행 방법**:
- **UI 영역 추출**: UI 설계서의 좌측 UI 영역(`<div class="ui-area">`) 내부 코드를 추출하고, 우측 명세 영역 및 circle-number 클래스는 제거
- **페이지 컴포넌트 생성**: `frontend/views/[menu-name]/` 디렉토리에 페이지별 `.js` 파일 생성
  - 목록 페이지: `List.js` (index.html 기반)
  - 등록/수정 통합: `Edit.js` (create.html, edit.html 기반 - 권장)
  - 또는 등록/수정 분리: `Create.js`, `Update.js`
- **표준 페이지 패턴 적용**: `frontend.composables.utils.instructions.mdc`의 표준 페이지 패턴(List, Edit, Create, Update, Detail) 사용
- **Composables 활용**: 다음 Composables 필수 사용
  - `useCrud`: CRUD 작업 통합 처리 (`frontend/composables/useCrud.js`)
  - `useAuth`: 인증 관련 로직 (`frontend/composables/useAuth.js`)
  - `useRouter`: 라우팅 헬퍼 (`frontend/composables/useRouter.js`)
  - `useSearch`: 검색 및 필터링 (`frontend/composables/useSearch.js`)
  - `useFormValidation`: 폼 유효성 검증 (`frontend/composables/useFormValidation.js`)
- **Utils 활용**: 공통 유틸리티 함수 사용
  - `frontend/utils/validators.js`: 유효성 검증 함수 (validateEmail, validatePassword, validatePhone 등)
  - `frontend/utils/constants.js`: 상수 정의 (USER_TYPES, ENGINE_TYPES 등)
  - `frontend/utils/formatters.js`: 데이터 포맷팅 함수
  - `frontend/utils/tableHelpers.js`: 테이블 헬퍼 함수 (createSortableColumn, createActionColumn 등)
- **ES6 모듈 변환**: HTML `<script>` 태그 내 Vue 코드를 ES6 모듈 형식으로 변환하고 `.js` 확장자 명시
  - CDN에서 로드된 전역 변수 사용: `const { ref } = typeof Vue !== 'undefined' ? Vue : {}`
  - Element Plus 전역 변수: `const ElMessage = typeof ElementPlus !== 'undefined' ? ElementPlus.ElMessage : {}`
- **스타일 분리**: 인라인 스타일을 `frontend/styles/main.css`로 이관 (필요시)
- **Mock 데이터 경로 수정**: API 엔드포인트 경로를 `frontend/api/v1/[menu-name]/` 구조에 맞게 변경
- **실제 API 연동 준비**: Backend API 개발 완료 후 Mock 데이터를 실제 API로 전환
  - `useCrud` 호출 시 `mockDataPath` 파라미터를 `null`로 설정
  - `apiEndpoint`는 `/api/users/vehicles` 형식으로 전달 (내부에서 자동 정규화)

### 3단계: 분할된 Frontend 소스코드 연동

**목표**: 분할된 컴포넌트와 페이지를 Vue Router와 연결하고, 전체 애플리케이션이 SPA 방식으로 동작하도록 통합

**수행 방법**:
- **인증 시스템 구성**:
  - `frontend/composables/useAuth.js`를 사용하여 로그인/로그아웃/토큰 관리 구현
  - `frontend/views/auth/Login.js`, `Register.js` 생성 (인증 페이지)
  - JWT 토큰은 `localStorage`에 저장 (`accessToken`, `refreshToken`, `user`)
  - `frontend/app.js`의 Axios 인터셉터에서 자동으로 토큰 추가 및 401 처리
- **레이아웃 컴포넌트 생성**:
  - `frontend/components/layout/AppLayout.js` 생성 (사이드바, 헤더 포함)
  - `frontend/views/Dashboard.js` 생성 (대시보드 화면)
  - 사이드바 메뉴 구성 및 네비게이션 연동
- **메뉴 관련 공통 모듈 (Task 기반 메뉴 구성)**:
  - **⚠️ 중요**: 메뉴는 Task 정의서에서 요구되는 분류에 따라 동적으로 구성합니다. 고정된 메뉴 구조를 사용하지 않습니다.
  
  **Task 정의서 기반 메뉴 분석**:
  1. **Task 정의서 메뉴 경로 추출**: 각 Task 정의서의 "메뉴 경로" 필드 확인
     - 예: `메뉴 경로: 사용자관리 > 회원가입/인증`
     - 예: `메뉴 경로: 차량관리 > 차량등록관리`
  2. **사용자 유형별 메뉴 분류**: Task 정의서의 "사용자 유형" 또는 "접근 권한" 섹션 확인
     - 각 Task가 어떤 사용자 유형(`CUSTOMER`, `BUSINESS`, `BUSINESS_ADMIN`, `SYSTEM_ADMIN` 등)에 접근 가능한지 파악
  3. **메뉴 구조 도출**: Task 정의서들을 분석하여 사용자 유형별로 접근 가능한 메뉴 목록 생성
     - 메뉴 경로의 상위 분류(예: "사용자관리", "차량관리")를 메뉴 그룹으로 구성
     - 각 메뉴 그룹 하위에 해당하는 Task들의 메뉴 경로를 하위 메뉴로 구성
  
  **AppLayout 컴포넌트 (`frontend/components/layout/AppLayout.js`)**:
  - 사이드바 메뉴: Task 정의서에서 추출한 사용자 유형별 메뉴 구조로 동적 구성
  - 메뉴 아이템 구조:
    ```javascript
    {
      title: '메뉴명',           // Task 정의서의 메뉴 경로에서 추출
      icon: 'ElementPlus 아이콘명', // 메뉴 성격에 맞는 아이콘 선택
      path: '/라우트 경로',        // Frontend 개발 결과 문서의 라우트 경로
      visible: true,              // 사용자 유형에 따라 표시 여부 결정
      userTypes: ['CUSTOMER']     // 접근 가능한 사용자 유형 배열 (선택사항)
    }
    ```
  - 사이드바 접기/펼치기 기능 (`sidebarCollapsed` 상태 관리)
  - 현재 활성 메뉴 하이라이트 (`activeMenu` 상태 관리)
  - 헤더: 페이지 제목 표시, 사이드바 토글 버튼, 알림 배지
  - **사이드바 레이아웃 구조**:
    - Flexbox 레이아웃 사용: `display: flex; flex-direction: column`으로 사이드바 구성
    - 사이드바 헤더: 상단 고정 (`flex-shrink: 0`)
    - 메뉴 영역: 중간 확장 가능 (`flex: 1; overflow-y: auto`) - 스크롤 가능
    - 사이드바 하단 사용자 정보 영역: 하단 고정 (`flex-shrink: 0`)
  - **사이드바 하단 사용자 정보 및 로그아웃 버튼**:
    - 위치: 사이드바 하단에 고정 배치 (메뉴와 겹치지 않도록 Flexbox 레이아웃 사용)
    - 사용자 프로필 정보 표시:
      - 사이드바가 펼쳐진 경우: 사용자 이름(굵게), 이메일(작은 글씨, 회색) 표시
      - 사이드바가 접힌 경우: 사용자 이름의 첫 글자로 아바타 표시 (`el-avatar` 사용)
    - 로그아웃 버튼:
      - 사이드바가 펼쳐진 경우: "로그아웃" 텍스트 표시
      - 사이드바가 접힌 경우: 아이콘 또는 "OUT" 텍스트 표시
      - 호버 시 툴팁으로 "로그아웃" 표시 (`title` 속성)
      - 클릭 시 확인 다이얼로그 후 로그아웃 처리 (`ElMessageBox.confirm` 사용)
    - 스타일: 배경색 `#263445`, 상단 테두리 `border-top: 1px solid #1f2d3d`
  
  **라우터 설정 (`frontend/router.js`)**:
  - 중첩 라우팅 구조: `AppLayout`을 부모 컴포넌트로 하고, 인증이 필요한 페이지는 `children`으로 구성
  - 인증 페이지 (`/auth/login`, `/auth/register`)는 `AppLayout` 외부에 배치
  - 인증 가드: `router.beforeEach`에서 인증 확인 및 리다이렉트 처리
  - 동적 라우트: `{ path: '/vehicles/:id/edit', component: VehicleEdit }`
  - **라우트 경로 매핑**: Task 정의서의 메뉴 경로와 Frontend 개발 결과 문서의 라우트 경로를 매핑
  
  **메뉴 동적 구성**:
  - `useAuth` composable에서 사용자 정보 로드 (`user.value?.userTypeCode` 또는 `user.value?.userType`)
  - Task 정의서에서 추출한 메뉴 구조를 기반으로 `menuItems` computed 속성으로 메뉴 동적 생성
  - 사용자 유형에 따라 접근 가능한 메뉴만 필터링하여 표시
  - 메뉴 클릭 시 `useRouter` composable의 `navigateTo` 함수로 페이지 이동

  **라우터 설정 (`frontend/router.js`)**:
  - **중첩 라우팅 구조 사용**: `AppLayout`을 부모 컴포넌트로 하고, 인증이 필요한 페이지는 `children`으로 구성
  - 인증 페이지 (`/auth/login`, `/auth/register`)는 `AppLayout` 외부에 배치
  - 예시 구조:
    ```javascript
    {
      path: '/dashboard',
      component: AppLayout,
      children: [
        { path: '', component: Dashboard },
        { path: '/user/profile', component: UserProfileEdit },
        // ... 기타 페이지
      ]
    }
    ```
  - 동적 라우트: `{ path: '/vehicles/:id/edit', component: VehicleEdit }`
  - **인증 가드 설정**: `router.beforeEach`에서 인증 확인
    - 인증되지 않은 사용자는 로그인 페이지로 리다이렉트
    - 이미 로그인된 사용자가 로그인 페이지 접근 시 대시보드로 리다이렉트
- **컴포넌트 등록**: `frontend/components/index.js`에 공통 컴포넌트를 export하여 중앙 관리 (선택사항)
- **API 모듈 설정**:
  - `frontend/app.js`: Axios 인터셉터 설정 (CDN으로 로드된 axios 사용)
    - 요청 인터셉터: JWT 토큰 자동 추가 (`Authorization: Bearer {token}`)
      - `localStorage.getItem('accessToken')`으로 토큰 가져오기
      - **모든 API 요청에 토큰 추가**: baseURL이 설정되어 있으므로 상대 경로(`/api/...`)와 절대 경로 모두 처리
      - 토큰이 존재하는 경우 모든 요청의 헤더에 자동으로 추가
    - 응답 인터셉터: 401 에러 시 로그인 페이지로 리다이렉트
      - 401 에러 발생 시 `localStorage`에서 토큰 및 사용자 정보 제거
      - `window.location.hash = '/auth/login'`으로 리다이렉트
  - `frontend/composables/useCrud.js`에서 API 호출 처리 (존재하는 경우)
    - **API 경로 정규화**: `apiEndpoint`에 `/api`가 포함된 경우 자동 제거 (baseURL에 이미 포함)
    - Mock 데이터는 `frontend/api/v1/[menu-name].json` 경로에서 로드
    - Backend API 개발 완료 후 실제 API 엔드포인트로 전환 시 `useCrud` 호출부에서 경로만 변경하면 됨
  - 직접 axios 호출 패턴 (useCrud 미사용 시):
    - `api.get('/[endpoint]')` 형식으로 직접 호출 (baseURL이 설정되어 있으므로 상대 경로 사용)
    - 인터셉터가 자동으로 JWT 토큰을 추가하므로 별도 헤더 설정 불필요
  - **Backend API 응답 구조 처리**:
    - **⚠️ 중요**: Backend API 응답은 `{result_code, result_message, data}` 형식만 사용합니다. `success` 필드는 없습니다.
    - Backend API는 다음 형식으로 응답:
      - 성공 (목록): `{result_code: "I0001", result_message: "...", data: {serviceCenters: [...]}}`
      - 성공 (단일): `{result_code: "I0001", result_message: "...", data: {serviceCenter: {...}}}`
      - 성공 (중첩 데이터): `{result_code: "I0001", result_message: "...", data: {operatingHours: [...], serviceItems: [...], equipment: [...]}}`
    - Frontend에서 응답 처리 시:
      - **성공 여부 확인**: `data.result_code?.startsWith('I')`로만 확인 (필수)
      - **에러 확인**: `result_code`가 "E"로 시작하면 에러
      - **`success` 필드 체크 금지**: `success` 필드는 Backend에서 제공하지 않으므로 체크하지 않음
      - 중첩된 데이터는 `data.data.serviceCenters`, `data.data.serviceCenter`, `data.data.operatingHours` 등으로 접근
      - 배열 데이터는 `Array.isArray()` 체크 후 사용
      - 대비용으로 `data.data`가 직접 배열인 경우도 처리
- **전역 상태 관리**: `frontend/store.js`에 전역 상태 설정 (선택사항)
  - 사용자 정보, 테마, 사이드바 상태 등
- **index.html 확인**: 메인 `frontend/index.html`에 Vue 앱 마운트 포인트 및 CDN 라이브러리 포함 확인
  - Vue 3, Vue Router, Element Plus, Element Plus Icons, Axios는 CDN으로 로드
  - ES6 모듈 방식으로 `app.js` 로드 (`type="module"`)
- **app.js 전역 에러 핸들러 구현**: `frontend/app.js`에 다음 에러 핸들러 필수 구현
  - **전역 에러 핸들러**: `window.addEventListener('error')`로 전역 에러 캐치 및 사용자 친화적 에러 메시지 표시
    - 에러 발생 시 콘솔에 로그 출력
    - `#app` 요소에 에러 메시지 표시 (에러 메시지, 브라우저 콘솔 확인 안내)
  - **Promise 거부 핸들러**: `window.addEventListener('unhandledrejection')`로 처리되지 않은 Promise 거부 캐치
    - 콘솔에 에러 로그 출력
  - **Vue 로드 확인**: CDN에서 Vue가 로드되었는지 확인 (`typeof Vue !== 'undefined'`)
    - Vue가 로드되지 않은 경우 `#app` 요소에 경고 메시지 표시
  - **모듈 로드 실패 처리**: `Promise.all()`로 모듈 로드 시도, 실패 시 에러 메시지 표시
    - `store.js`, `router.js` 동적 import 실패 시 사용자에게 에러 메시지 표시
    - 에러 메시지, 스택 트레이스, 브라우저 콘솔 확인 안내 포함
- **경로 테스트**: 브라우저에서 라우팅 동작 및 페이지 전환 확인 (사용자에게 테스트 요청)

## 추가 고려사항

### **인증 및 보안**
- JWT 토큰은 `localStorage`에 저장 (`accessToken`, `refreshToken`, `user`)
- `frontend/app.js`의 요청 인터셉터에서 자동으로 토큰 추가
- 401 에러 발생 시 자동으로 로그인 페이지로 리다이렉트
- 라우터 가드에서 인증 상태 확인 및 리다이렉트 처리

### **API 연동**
- `frontend/app.js`: Axios 인터셉터 설정 (CDN으로 로드된 axios 사용)
- 직접 axios 호출: `api.get('/[endpoint]')` 형식 (baseURL이 설정되어 있으므로 상대 경로 사용)
- `useCrud` 사용 시: API 경로 정규화 (`/api`로 시작하는 경로는 자동으로 제거)
- **백엔드 응답 형식 및 처리**:
  - **⚠️ 중요**: Backend API 응답은 `{result_code, result_message, data}` 형식만 사용합니다. `success` 필드는 없습니다.
  - 성공 (목록): `{result_code: "I0001", result_message: "...", data: {items: [...]}}`
  - 성공 (단일): `{result_code: "I0001", result_message: "...", data: {item: {...}}}`
  - 성공 (중첩 데이터): `{result_code: "I0001", result_message: "...", data: {serviceCenter: {...}, operatingHours: [...], serviceItems: [...], equipment: [...]}}`
  - 성공 (작업): `{result_code: "I0007", result_message: "...", data: {...}}`
  - 에러: `{result_code: "E1001", result_message: "..."}`
- **응답 데이터 접근 패턴**:
  - **성공 여부 확인**: `data.result_code?.startsWith('I')`로만 확인 (필수)
  - **에러 확인**: `result_code`가 "E"로 시작하면 에러
  - **`success` 필드 체크 금지**: `success` 필드는 Backend에서 제공하지 않으므로 체크하지 않음
  - 중첩된 데이터는 `data.data.serviceCenters`, `data.data.serviceCenter` 등으로 접근
  - 배열 데이터는 `Array.isArray()` 체크 후 사용
  - 대비용으로 `data.data`가 직접 배열인 경우도 처리

### **메뉴 관련 공통 모듈 (Task 기반 메뉴 구성)**
- **⚠️ 중요**: 메뉴는 Task 정의서에서 요구되는 분류에 따라 동적으로 구성합니다. 고정된 메뉴 구조를 사용하지 않습니다.
- **Task 정의서 분석**: 각 Task 정의서의 "메뉴 경로" 및 "사용자 유형" 정보를 추출하여 메뉴 구조 도출
- **AppLayout 컴포넌트**: Task 정의서에서 추출한 메뉴 구조로 사용자 유형별 동적 메뉴 구성
  - 메뉴 경로의 상위 분류를 메뉴 그룹으로 구성
  - 각 메뉴 그룹 하위에 해당하는 Task들의 메뉴 경로를 하위 메뉴로 구성
  - 사용자 유형에 따라 접근 가능한 메뉴만 필터링하여 표시
- **라우터 설정**: 중첩 라우팅 구조, 인증 가드, 동적 라우트
- **메뉴 동적 구성**: `useAuth` composable에서 사용자 정보 로드 후 Task 기반 메뉴 동적 생성

### **레이아웃 구조**
- `AppLayout`: 사이드바, 헤더, 메인 컨텐츠 영역 포함
  - 사이드바: Flexbox 레이아웃으로 헤더(고정), 메뉴(확장 가능, 스크롤), 하단 사용자 정보(고정) 구성
  - 사이드바 하단: 사용자 프로필 정보(이름, 이메일 또는 아바타) 및 로그아웃 버튼 고정 배치
  - 사이드바 접힘 상태에서도 사용자 정보(아바타) 및 로그아웃 버튼 표시
- 중첩 라우팅: 인증이 필요한 페이지는 `AppLayout`의 `children`으로 구성
- 대시보드: 통계 카드 및 빠른 작업 메뉴 제공

### **CDN 라이브러리 사용**
- Vue 3, Vue Router, Element Plus는 CDN으로 로드
- 전역 변수로 접근: `Vue`, `VueRouter`, `ElementPlus`, `ElementPlusIconsVue`
- ES6 모듈에서 전역 변수 사용 시 안전 체크: `typeof Vue !== 'undefined' ? Vue : {}`

### **전역 에러 핸들러**
- `app.js`에 전역 에러 핸들러 필수 구현
- `window.addEventListener('error')`: 전역 에러 캐치 및 사용자 친화적 메시지 표시
- `window.addEventListener('unhandledrejection')`: 처리되지 않은 Promise 거부 캐치
- Vue/CDN 로드 확인: 라이브러리 로드 실패 시 경고 메시지 표시
- 모듈 로드 실패 처리: 동적 import 실패 시 에러 메시지 및 스택 트레이스 표시
- 모든 에러 핸들러는 콘솔 로그와 함께 사용자에게 명확한 안내 메시지 제공
