## 📋 변수 정의

UI 설계서 파일명과 경로에 사용되는 변수:

- **{{ systemName }}**: 파트너 시스템, 파트너 포털, 파트너 플랫폼 관리자 시스템, 파트너 플랫폼 전체공통, 글로벌 OMS, 브랜드원 시스템 기능 고도화 중 하나
- **{{ 1depthMenuName }}**: 요구사항 구조를 파악하여 1Depth 메뉴명 정의 (예: 기준정보관리)
- **{{ 2depthMenuName }}**: 요구사항 구조를 파악하여 2Depth 메뉴명 정의 (예: 성분관리)
- **{{ 3depthMenuName }}**: 요구사항 구조를 파악하여 3Depth 메뉴명 정의 (선택, 예: 전성분관리)
- **{{ 3digitSeqNum }}**: 먼저 만들어진 파일명의 패턴을 파악하여 순번 채번 (예: 001, 002, 003)
- **{{ fileName }}**: `{{ systemName }}-{{ 3digitSeqNum }}.{{ 1depthMenuName }}-{{ 2depthMenuName }}-{{ 3depthMenuName }}`

## 📝 실행 프로세스

### 1단계: DB 스키마 기반 Mock 데이터 생성 및 보완

**목표**: UI 설계서에서 사용할 Mock 데이터를 Task 정의서와 DB 스키마를 기반으로 생성

**수행 방법**:
1. Task 정의서에서 필요한 데이터 항목 파악
2. **데이터 모델 분석** (필수)
   - **스키마 구조 파악**:
     - `database/schemas/*.sql` 파일 중 스키마 생성 관련 파일(`create_schema*.sql` 등)을 참조하여 데이터베이스 스키마 생성 및 구조 파악
   - **권한 구조 파악**:
     - `database/schemas/*.sql` 파일 중 권한 관련 파일(`*role*.sql`, `*user*.sql`, `*permission*.sql`, `grant*.sql` 등)을 참조하여 역할(Role), 사용자(User), 권한 부여 구조 파악
   - **테이블 구조 및 관계 파악** (모든 테이블 생성 파일 참조):
     - `database/schemas/*.sql` 파일 중 테이블 생성 관련 파일(`create_tables*.sql`, `create_table*.sql` 등)을 모두 참조하여 테이블 구조와 관계 파악
     - 파일명 패턴 예시: `create_tables_phase*.sql`, `create_table_*.sql` 등
     - 각 Phase 파일은 기능 도메인별로 테이블을 구성하므로, 파일명과 내용을 분석하여 도메인별 데이터 모델 이해
   - **목적**: 
     - Mock 데이터 생성 시 전체 데이터베이스 구조 파악
     - 기존 테이블 및 관계 활용 여부 판단
     - 스키마 및 권한 구조를 고려한 Mock 데이터 설계
     - Phase별/도메인별 테이블 구조를 통한 기능 도메인별 데이터 모델 이해
3. `docs/02.design/02.ui/common/api/v1/[resource].json` Mock 데이터 파일 확인
   - 파일이 있으면 사용
   - 없으면 생성 (실제 Backend API 응답 형식과 동일하게)
   - **파일 형식**: `.json` 확장자를 포함하여 저장 (예: `brand-masters.json`, `users.json`)
   - **응답 구조**: Backend API와 동일한 JSON 형식

**Mock 데이터 구조**: 지침 파일 섹션 6.2 참조

### 3단계: UI 설계서 검증 및 보완

**목표**: 작성된 UI 설계서가 모든 요구사항을 충족하는지 검증

**검증 항목**:

1. **지침 준수 검증**
   - [ ] UI 설계서 작성 지침 (`.cursor/rules/frontend.ui-design.instructions.mdc`) 준수
   - [ ] UI/UX 스타일링 지침 준수
   - [ ] Element Plus 컴포넌트 사용 규칙 준수
   - [ ] 기존 UI 설계서와 일관성 (아이콘, 컴포넌트 스타일, 옵션 등)

2. **Task 정의서 반영 검증**
   - [ ] Task 정의서의 모든 기능 요구사항이 UI로 표현됨
   - [ ] 모든 인터랙션 요소에 번호 부여
   - [ ] 우측 명세에 각 요소의 상세 설명 작성
   - [ ] 누락된 내용 없이 완전히 반영

3. **데이터 및 동작 검증**
   - [ ] Mock 데이터가 DB 스키마와 일치
   - [ ] Mock 데이터 파일이 `api/v1/[resource].json` 경로에 저장됨
   - [ ] Element Plus 컴포넌트만 사용
   - [ ] VSCode Live Server로 정상 동작 확인 가능

4. **제약사항 준수 검증**
   - [ ] inline style 미사용
   - [ ] 금지된 버튼/기능 미포함
   - [ ] 버튼 배치 규칙 준수
   - [ ] 페이지네이션 layout 올바르게 정의

**보완 작업**:
- 검증 결과 발견된 문제점 즉시 수정
- Task 정의서 내용 누락 시 추가 반영
- 일관성 문제 발견 시 기존 UI 설계서 패턴으로 조정
