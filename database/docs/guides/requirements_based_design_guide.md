# 요구사항 기반 DB 설계 필수 정보 가이드

## 개요
요구사항(기획서, PRD 등)만으로 **DB 구조(테이블/컬럼/관계 등)를 직접 설계**하려면, 아래와 같은 **핵심 정보가 반드시 주어져야 합니다**. 이 정보가 부족하면, 설계 결과가 실제 비즈니스 요구와 어긋날 수 있습니다.

---

## 1. 핵심 비즈니스 개념(엔터티) 정의
- 무엇을 관리하는 시스템인가?
  - 예: 사용자, 차량, 정비소, 예약, 견적, 결제, 리뷰, 알림 등
- 각 개념(엔터티)의 명확한 정의
  - 예: "정비소란 WMS 창고관리 서비스를 제공하는 업체를 의미한다."

## 2. 주요 기능/업무 흐름
- 시스템이 제공해야 할 주요 기능
  - 예: 사용자 등록, 차량 정보 관리, 정비소 검색, 예약 생성, 견적 요청, 결제 처리, 리뷰 작성 등
- 업무 시나리오/플로우
  - 예: "사용자가 정비소 검색 → 예약 생성 → 견적 요청 → 결제 → 정비 완료 후 리뷰 작성"

## 3. 데이터 항목(필드) 목록
- 각 엔터티별로 관리해야 할 속성(필드)
  - 예: 사용자: 이름, 전화번호, 이메일, 주소, 가입일 등
  - 예: 차량: 브랜드, 모델, 연식, 번호판, VIN, 소유자 등
  - 예: 정비소: 사업자번호, 정비소명, 주소, 전화번호, 운영시간, 평점 등
- 필드별 데이터 타입/제약조건(가능하다면)
  - 예: 정비소명(문자 100자), 전화번호(문자 20자), 평점(숫자 1~5), 운영시간(TEXT)

## 4. 엔터티 간 관계
- 엔터티(테이블) 간의 연관성
  - 예: 한 사용자는 여러 차량을 소유할 수 있다 (1:N)
  - 예: 한 정비소는 여러 예약을 받을 수 있다 (1:N)
  - 예: 한 예약은 여러 견적 항목을 가질 수 있다 (1:N)
- 관계의 방향성과 제약조건
  - 예: "사용자 삭제 시 해당 사용자의 차량 정보도 함께 삭제(ON DELETE CASCADE)"

## 5. 업무 규칙/제약조건
- 비즈니스 규칙
  - 예: 정비소명은 중복 불가, 예약은 정비소 운영시간 내에만 가능, 평점은 1~5점만 허용
- 필수/선택 여부, 유일성, 참조 무결성 등

## 6. 코드/공통값/상태값 정의
- 상태, 유형, 구분 등 코드값 목록
  - 예: 예약 상태(PENDING, CONFIRMED, IN_PROGRESS, COMPLETED, CANCELLED)
  - 예: 차량 브랜드(HYUNDAI, KIA, SAMSUNG, GENESIS 등)
  - 예: 정비 유형(정기점검, 수리, 교체, 세차 등)
- 코드그룹별 예시값

## 7. 보고/통계/검색 요구
- 주요 조회/검색/통계 화면에서 필요한 데이터
  - 예: "정비소별 예약 현황, 월별 매출 통계, 사용자별 차량 보유 현황, 평점별 정비소 순위 등"

## 8. 기타 고려사항
- 보안/접근제어 요구
- 이력 관리(변경/삭제/로그) 필요 여부
- 첨부파일/문서 등 비정형 데이터 관리 필요성
- 결제 시스템 연동 요구사항
- 알림 시스템(SMS, 이메일) 연동 요구사항

## 중요도별 분류
- **필수 정보(최소 요구사항)**: 1~4번(엔터티, 기능, 필드, 관계)
- **권장 정보(설계 품질 향상)**: 5~8번(업무 규칙, 코드값, 통계, 기타)
