## 1. 개요

### 1.1 목적

운영 중인 WMS의 안정적 운영을 위해 기존 기능을 개선·보완한다. 고객 요청 반영, 오류·예외 상황 완화, 사용자 오입력·오동작 최소화, 상태 변경 및 처리 흐름의 가시성 확보, 유사 요청 대응력 강화를 목적으로 한다. **핵심 업무 프로세스(입고 주문 생성 → 검수 → 입고 확정 → 재고 이동 → 출고 주문 생성 → 피킹 → 출고 확정)의 흐름 자체는 변경하지 않는다.**

### 1.2 범위

**포함**

- 신규 화면 추가(일부), 기존 화면 일부 수정
- 입·출고 주문 생성 배치 로직 보완 및 안정성 강화
- 내부 처리 로직 및 인터페이스 개선(저장 프로시저 처리 흐름 정리, 상태 변경 로직 명확화 등)
- 화면 사용성 개선, 오류 가능성이 높은 입력 보완, 상태 변경 시 사용자 인지 강화
- 배치 예외 시 오류 원인 식별 개선, 재처리 시 중복 처리 가능성 최소화
- 이력·로그(부분 이력, 개인정보 화면 조회/다운로드 이력, 운영 대응 목적 추적) 관련 요구사항

**제외**

- 신규 업무 프로세스 추가
- 대규모 시스템 구조 변경

**개발·스키마 경계(PR 원문)**

- PRD에 따르면 **테이블 구조 변경은 운영자가 수행**하고, PRD 기반 Cursor AI 개발 범위에는 **테이블 변경이 포함되지 않는다.**
- 본 문서는 **물리 DDL을 정의하지 않는다**(섹션 8 참고).

**PRD와 본 저장소(교육 프로젝트)의 관계**

- PRD는 운영 레거시 전제(Nexacro, MSSQL, 세션 인증, REST 비사용 등)를 포함한다.
- 본 저장소(`skax-devlab-wms`)의 **구현 기술 스택·인증 방식**은 프로젝트 공통 지침(예: Spring Boot, Vue 3, PostgreSQL, JWT)을 따르며, **업무·품질 요구는 PRD를 우선**으로 한다. 상세는 §3.2, §6.2, §7.1.

## 2. 핵심 기능 요구사항

### 2.1 화면 개선 및 사용자 인지 (REQ-WMS-IMPR-001)

#### 2.1.1 기본 기능

- 기존 화면의 사용성을 개선한다.
- 오류 발생 가능성이 높은 입력 항목을 보완한다.
- **상태 변경**과 연관된 버튼·기능에 대해 사용자 인지를 강화한다.  
  상태 변경에는 입고 검수·취소·삭제, 출고 지시·취소·삭제 등 WMS 관련 **모든 변경**이 포함된다.
- 잘못된 조작 시 즉시 확인 가능한 피드백을 제공한다.
- 상태 변경이 발생한 경우 사용자가 **명확히 인지**할 수 있어야 한다.

#### 2.1.2 비즈니스 규칙

- 화면은 PRD의 기술 전제(예: 화면→컨트롤러→서비스→맵퍼, Map 기반 전달)를 **레거시 참고**로 두되, 실제 구현은 본 프로젝트 구조에 맞는다.
- 상태 전환 조건은 **기존 업무 규칙**을 따른다. 불필요한 상태 역행은 허용하지 않는다.

### 2.2 입·출고 주문 생성 배치 (REQ-WMS-IMPR-002)

#### 2.2.1 기본 기능

- 입·출고 주문 생성은 **정기 배치**로 수행한다.
- **수동 배치 실행 기능은 제공하지 않는다.** 다만 로컬 환경 테스트·디버깅 목적에 한해 예외를 둔다(PR 원문).
- 배치 재처리 기능은 기존에 존재하며, **본 개선에서는 재처리 흐름을 유지**한다.

#### 2.2.2 비즈니스 규칙

- 배치 처리 중 오류 발생 시 **오류 원인 식별이 가능**하도록 개선한다.
- 재처리 시 **중복 처리 가능성을 최소화**한다.
- 외부에서 입·출고 요청 데이터가 수신되고, 정기 배치로 WMS 입·출고 주문이 생성되는 흐름을 유지한다.

### 2.3 상태 변경 및 데이터 정합성 (REQ-WMS-IMPR-003)

#### 2.3.1 기본 기능

- PRD 기준: 상태 변경은 **저장 프로시저 내에서 데이터 저장과 함께** 처리되며, 상태 변경만 단독 수행하는 구조는 사용하지 않는다.

#### 2.3.2 비즈니스 규칙

- 상태 변경 중 오류 발생 시 **데이터 불일치가 발생하지 않도록** 한다.

### 2.4 이력·로그(부분 이력·추적) (REQ-WMS-IMPR-004)

#### 2.4.1 기본 기능

- **모든 상태 변경**에 대해 이력을 기록하지는 않으며, **일부 주요 단계**에 대해서만 이력 테이블을 사용한다(PR 원문).
- **개인정보 포함 화면**에 대해서는 조회 및 다운로드 이력을 관리한다(PR 원문).
- 고객 문의 대응을 위해 **상태 변경 흐름 추적이 가능**하도록 고려한다(운영 로그 관점).

#### 2.4.2 비즈니스 규칙

- “주요 단계”의 구체 목록은 PRD에 **미기재**이므로, 운영·보안 정책 및 별도 `docs/01.analysis/02.requirement/*.md`로 확정할 때 반영한다.

### 2.5 보안·권한 (REQ-WMS-IMPR-005)

#### 2.5.1 기본 기능

- PRD: 자체 로그인, **세션 기반 인증** 기술 전제.
- 권한 유형: 물류센터 사용자, 화주 사용자, 센터 관리자, IT 관리자. **기존 권한 정책 유지**.

#### 2.5.2 비즈니스 규칙

- 역할·권한에 따른 기능 노출·처리 제한은 기존 정책을 따른다.

## 3. 명확화된 정책 및 규칙

### 3.1 PRD에서 직접 확정된 정책

- PRD에 명시된 범위·제외·배치 방식·재처리 유지·이력 부분 기록·PII 화면 이력·데이터 정합성 등은 **본 문서 §2에 반영된 그대로** 요구사항이다.

**구현방안**: 요구사항 구현 시 §2의 REQ ID를 추적 가능하도록 설계·테스트 범위에 명시한다.

### 3.2 “주요 단계” 이력 대상(PR 미기재)

- PRD는 일부 주요 단계만 이력 테이블 사용을 명시하나, **대상 단계 목록은 PRD에 없음**.

**구현방안**: `docs/01.analysis/02.requirement/`에 별도 `.md`로 목록을 확정하거나, 운영 협의 후 요구사항정의서를 개정한다. 확정 전까지는 **이력 설계를 구체 테이블 수준으로 고정하지 않는다.**

### 3.3 개인정보 포함 화면 범위(PR 목록 미첨부)

- PRD는 PII 화면에 대한 조회·다운로드 이력을 요구하나, **화면 ID 목록은 PRD에 없음**.

**구현방안**: `02.requirement` 보완 문서 또는 보안 담당자와 화면 목록을 합의한 뒤 §2.4에 반영한다.

### 3.4 PRD 기술 스택 vs 본 교육 저장소

- PRD: Nexacro, MSSQL, 세션, REST 비사용(레거시 서술), Map·SP 중심.
- 본 저장소 프로젝트 공통 지침: Vue 3, Spring Boot, PostgreSQL, JWT 등.

**구현방안**: **업무 규칙·화면·배치·품질 요구는 PRD 우선**, **표현 계층·API·DB 제품·인증 구현 방식은 프로젝트 공통 지침**에 따른다. 레거시 용어는 분석·대응용 참고로 유지한다.

## 4. 예외 처리 및 오류 시나리오

오류 코드 구간(예시 할당):

- **E1000–E1999**: 배치·연계·재처리(중복·재시도·실패 건 식별)
- **E2000–E2999**: 상태 전환·업무 규칙 위반(역행 불가 등)
- **E3000–E3999**: 입력 검증·오입력·선행 조건 미충족
- **E4000–E4999**: 인증·권한

### 4.1 배치 처리 실패

- **예외**: 입·출고 주문 생성 배치가 특정 건에서 실패한다.
- **오류** → `"배치 처리 중 오류가 발생했습니다. 건별 사유를 확인하세요."` (예: **E1001**)

### 4.2 재처리 시 중복 처리 위험

- **예외**: 동일 연계 키·참조번호에 대해 재처리가 중복 반영될 수 있음.
- **오류** → `"이미 처리된 건입니다. 재처리 조건을 확인하세요."` (예: **E1002**)

### 4.3 허용되지 않는 상태 전환

- **예외**: 현재 상태에서 요청한 상태 변경이 업무 규칙상 불가.
- **오류** → `"현재 상태에서 수행할 수 없는 작업입니다."` (예: **E2001**)

### 4.4 상태 변경 처리 중 오류

- **예외**: 처리 중 서버·DB 오류로 트랜잭션이 롤백되거나 불완전 종료.
- **오류** → `"처리 중 오류가 발생했습니다. 데이터 정합성을 확인한 후 다시 시도하세요."` (예: **E2002**)

### 4.5 입력값 오류

- **예외**: 필수값 누락, 형식 오류.
- **오류** → `"입력값을 확인하세요."` (예: **E3001**)

### 4.6 권한 없음

- **예외**: 사용자 역할이 해당 기능을 수행할 수 없음.
- **오류** → `"권한이 없습니다."` (예: **E4001**)

## 5. 테스트 시나리오

### 5.1 정상 시나리오

- 입·출고 연계 데이터가 존재할 때 정기 배치가 **정상적으로 WMS 주문을 생성**한다.
- 허용된 상태 전환만 수행되며, 성공 시 사용자에게 **상태 변경이 인지 가능한 피드백**이 제공된다.
- 재처리는 기존 흐름을 유지하며, **중복 없이** 또는 정책에 따라 **정상 스킵**된다.

### 5.2 예외 시나리오

- 배치 오류 건에 대해 **원인 식별 가능한 정보**(로그·메시지·건식별자)가 남는다.
- 허용되지 않는 상태 전환 요청 시 **E2001** 등으로 거부되고 데이터가 불일치하지 않는다.
- 권한 없는 사용자는 **E4001**로 거부된다.
- PII 화면이 확정된 경우, 조회·다운로드 이력이 정책대로 남는다(목록 확정 후).

## 6. 성능 및 보안 요구사항

### 6.1 성능 요구사항

- PRD에 **응답 시간·처리량·동시 사용자 수 등의 수치 목표는 미정의**이다.
- 운영 중단 최소화(PR §15.3)와 배치 안정성 강화(PR §4.2, §5.3)를 **정성 목표**로 둔다.

**구현방안**: 수치 목표가 필요하면 `02.requirement` 또는 별도 비기능 요구로 확정 후 본 문서를 개정한다.

### 6.2 보안 요구사항

- PRD: 자체 로그인, 세션 기반(레거시 기술 서술), PII 화면 조회·다운로드 이력.
- 본 저장소 구현 시 **JWT 기반 교육용 인증**을 사용하는 경우, **역할·권한 분리**는 §2.5의 유형을 반영한다.

**구현방안**: 세션 vs JWT의 **구현 차이**는 프로젝트 표준에 따르되, **접근 통제·감사(PII 이력)** 요구는 유지한다.

---

## 7. 시스템 연동 요구사항

### 7.1 필수 연동 시스템

- **외부 시스템**: 입·출고 주문(요청) 데이터는 외부 시스템에서 전달된다(일부 주문은 사용자 직접 생성).
- **내부 배치**: 정기 배치가 **수신 데이터를 기반으로 WMS 입·출고 주문을 생성**한다.

**구현방안**: 연계 채널(파일·API·DB 등)의 **구체 규격은 PRD에 미정의**이므로, 교육/개발 환경에서는 목업·스텁·테이블 적재 등으로 대체할 수 있으나, **업무 흐름(외부→배치→WMS 주문)** 은 유지한다.

## 8. 데이터 모델 요구사항 [물리 모델 생성 금지]

### 8.1 핵심 엔티티(개념)

PRD 및 도메인 서술에 기반한 **논리 엔티티**만 기술한다. 테이블·컬럼 DDL은 **요구사항 정의 범위에 포함하지 않는다.**

| 개념 영역 | 엔티티(예시) | 설명 |
|-----------|--------------|------|
| 사용자·권한 | 사용자, 사용자그룹 | 로그인·역할·권한 부여 |
| 입고 | 입고 헤더, 입고 상세 | 입고 주문·상태·검수·확정 |
| 출고 | 출고 헤더, 출고 상세 | 출고 주문·피킹·확정 |
| 재고 | 로트, 재고(셀), 품목별 재고 | 수량·위치·가용성 |
| 공통 | 공통 코드(상태값 등) | 상태 전환·표시 |

PRD에 예시로 등장한 물리 테이블명(TWMS_* 등)은 **레거시 참고**이며, 본 교육 DB의 실제 테이블명·스키마는 **프로젝트 DB 명명 규칙**에 따른다.

### 8.2 데이터 운영 고려사항

- **테이블 변경은 운영자 수행**, PRD 기반 Cursor 개발 범위에 **DDL 미포함**(PR 원문).
- **데이터 정합성 유지 필수**, 이력 데이터 관리 기준 유지(PR §14.3).
- **“주요 단계” 이력 대상·PII 화면 목록**이 확정되기 전에는 이력·감사 설계를 **완결하지 않는다**(§3.2, §3.3).
